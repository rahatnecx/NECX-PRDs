2. Executive Summary

NECX Drive is a Google-Drive-style cloud file storage feature embedded in the NECX platform, scoped to a single agency (service center). It lets agency staff upload, organise, browse, share, and track files without leaving the VS360 web application. Files are stored in Azure Blob Storage and streamed directly between the browser and Azure using short-lived signed URLs, so file bytes never transit the NECX API servers on upload or download.

NECX serves victims and survivors of crime and the agencies and advocates who support them. Advocates routinely handle sensitive supporting documentation — identity documents, police reports, court paperwork, benefit applications, photographs — that today are scattered across personal drives, email attachments, and ad-hoc shared folders. NECX Drive gives each agency a single, tenant-isolated, auditable place to keep this material, with per-colleague and agency-wide sharing, expiring share links, soft-delete/restore via a Trash, and storage-usage visibility.

The feature is designed around familiar cloud-drive conventions: a left rail of views (My Drive, Shared with me, Agency Drive, Starred, Trash), breadcrumb navigation, drag-and-drop upload (including whole folders), a per-row kebab action menu, MIME-typed file icons, and a storage meter. Security is foundational: strict multi-tenant isolation on every query, server-constructed blob paths that make path traversal impossible, MIME whitelisting plus magic-byte sniffing at commit, a 250 MB file cap enforced at three layers, SHA-256-hashed share-link tokens, and a full audit trail on every mutating action.

NECX Drive is a VS360-web-only feature in this release: there is no Victim-app or Workspace-app surface. It complements — but is distinct from — the E-Sign / Document Center feature (case-scoped documents proxied through the API); NECX Drive is general-purpose agency file storage with direct-to-Azure transfer.

3. Product Overview & Vision

3.1 Problem Statement

Agency staff need to store and share operational files (case supporting documents, templates, agency records, media) in a way that is centralised, tenant-isolated, access-controlled, and auditable. Absent a first-party solution, staff resort to personal cloud drives and email, which fragments records, breaks the audit trail, risks exposing sensitive victim material outside the agency's control, and provides no storage governance or usage visibility.

3.2 Product Vision

Provide every NECX agency with a secure, self-service, Google-Drive-class file store built into VS360 — one that feels immediately familiar (folders, breadcrumbs, drag-drop, share dialogs, Trash) while enforcing NECX's tenancy, safety, and auditing guarantees underneath. Files move directly between the browser and Azure Blob Storage for performance and to keep sensitive bytes off application servers, and every share, download, and deletion is recorded.

3.3 Position in the NECX Ecosystem

NECX is a single unified backend (Node.js/Express, Objection/Knex over PostgreSQL, Azure Blob Storage) serving four clients: NECX Admin, VS360 web, Workspace mobile, and the Victim mobile app. NECX Drive is delivered exclusively through VS360 web for agency staff. It reuses the platform's authentication (req.issuer), context gate (allowOnly('VS360')), multi-tenancy (service_center_id), and audit infrastructure (logAudit / ActionLogs). It is adjacent to the E-Sign / Document Center feature (case-attached documents) but architecturally distinct: NECX Drive owns its own Azure Blob adapter and uses direct-to-Azure SAS transfer rather than proxying bytes through Express.

4. Goals, Objectives & Success Metrics

4.1 Goals & Objectives

  • Give agency staff a familiar, self-service file store (folders, upload, browse, share, Trash) inside VS360.
  • Keep sensitive file bytes off application servers by transferring directly between browser and Azure via short-lived signed URLs.
  • Enforce strict multi-tenant isolation so no agency can ever see or reach another agency's files.
  • Provide flexible, revocable sharing: with specific colleagues, with the whole agency, and via expiring links.
  • Give agencies visibility into storage consumption through a usage meter and a per-category storage breakdown.
  • Maintain a complete audit trail of security-relevant actions (download, delete, restore, share, link, copy, empty-trash).
  • Prevent abuse and accidental data loss through rate limiting, size caps, MIME validation, and soft-delete with restore.

5. Scope

5.1 In Scope

  • Agency-scoped folder/file model with a self-referencing folder tree and files at root or within a folder.
  • Folder creation with case-insensitive per-parent name uniqueness (auto-suffixing on collision).
  • Two-phase, direct-to-Azure file upload: write-SAS issuance, browser PUT to Azure, then metadata commit with size and MIME verification.
  • Browsing views: My Drive, Shared with me, Agency Drive, Starred, Trash — with breadcrumb navigation, pagination, and sorting in My Drive.
  • Move, rename, and star/unstar for both folders and files (with folder circular-move protection).
  • Soft-delete to Trash, restore (with re-parent-to-root safety), and empty-trash (hard delete + Azure blob purge).
  • Sharing: with specific same-agency users, with the whole agency, and via revocable expiring share links (token shown once, stored only as a SHA-256 hash).
  • Copy a shared or owned file into the caller's My Drive (Azure server-side copy).
  • Filename search within a view (trigram-indexed ILIKE, with LIKE metacharacter escaping).
  • Storage-usage tracking (per-agency running counter surfaced on every list) and a per-MIME-category storage breakdown.
  • File-type icons (MIME → coloured icon) in the VS360 UI.
  • Direct-to-Azure download via short-lived read SAS (returned as JSON, opened by the client).
  • Full audit trail on mutating/security-relevant actions.

5.2 Out of Scope

  • Hard storage quota enforcement. There is no per-agency max_bytes limit checked before upload. drive_storage_usage is a tracked running counter only; the UI displays a 15 GB reference figure, but it is not enforced server-side (see Risks). The only server-side abuse guard is the SAS rate limiter.
  • In-app file preview/rendering. Files open in a new browser tab via the read-SAS URL; there is no embedded viewer.
  • Victim-app and Workspace-app surfaces. NECX Drive is VS360-web-only; the entire router is gated by allowOnly('VS360').
  • Real-time collaboration / sockets. No socket events; UI feedback is via local toasts. No live co-editing or presence.
  • Automatic Trash retention purge worker. UI copy states items auto-delete after 30 days, but only manual empty-trash currently hard-deletes and decrements usage; a background cleanup worker is referenced as a future phase.
  • Cross-agency sharing. Sharing is limited to same-agency users or the whole agency.
  • File versioning / revision history of file contents.
  • Cross-user move. Ownership stays with the creator; non-owners get "Save a copy to My Drive" instead of edit rights.

7. System Context & Architecture

7.1 Technology Stack

  • Backend: Node.js + Express (ESM), Objection/Knex over PostgreSQL. Drive module under src/apps/drive/ (route, controller, Joi validator, ~1900-line service, repository), with constants in src/lib/constants/drive.js and env config in src/config/drive.upload.config.js.
  • Storage: Azure Blob Storage in staging/production; Azurite emulator in dev/docker. The only module importing @azure/storage-blob is src/services/v2/shared/storage/drive/azureBlobAdapter.js.
  • Frontend: VS360 web (React/Vite + Mantine + TanStack Query). Feature under src/pages/Drive/ with a useDrive.js hook layer and src/services/drive.js axios wrappers.
  • Database: PostgreSQL with the pg_trgm extension (for trigram filename search). Five tables under migrations 20260502100000..100400.

7.2 How It Talks to the Backend

Every route is mounted under /web (e.g. GET /web/drive/list) and passes through the standard middleware chain: rate-limiter → auth (populates req.issuer with id, uid, roleInfo.context, serviceCenterInfo) → allowOnly('VS360') → Joi validator → controller → service → repository. Success responses use the envelope { status, msg, data }; validation failures return 400 { status, msg } (Joi abortEarly:false, stripUnknown). The VS360 axios factory baseURL already includes the /web prefix.

7.3 Direct-to-Azure Transfer Model

Uniquely among NECX file features, Drive transfers bytes directly between the browser and Azure, never through Express:

  • Upload is two-phase: the API issues a 10-minute write SAS for a server-constructed blob path; the browser PUTs the file bytes straight to the SAS URL (using a fresh axios instance so NECX auth headers do not leak to Azure); the API then commits metadata after verifying the blob's size and sniffing its MIME.
  • Download returns a 5-minute read-SAS URL as JSON (not an HTTP 302). Because the platform uses Authorization: Bearer headers, a redirect would drop auth; instead the client fetches the signed URL over authenticated AJAX and then navigates the browser to the SAS URL.
  • Copy uses Azure server-side blob copy, so bytes never transit Express even when duplicating a file.

7.4 Real-time / Notifications & Tenancy

NECX Drive uses no sockets; user feedback is via local toasts (upload progress, success/error). Multi-tenancy is enforced by service_center_id on every query; foreign-tenant rows are treated as "not found" to avoid existence leaks. All mutating/security-relevant actions write an audit record.

8. Functional Requirements

8.1 Folder & File Model

Folders form a per-agency self-referencing tree: each folder has an optional parent_folder_id (null = root). Files belong to exactly one folder or sit at root (folder_id null). Both entities carry owner_user_id, service_center_id, name, is_starred, and deleted_at (soft-delete). Each file additionally stores mime, size_bytes, and blob_name (the Azure object key). External identifiers are UUID v4 uid values; a file's uid equals the trailing segment of its blob path.

FR-MODEL-02Files shall reference an optional folder_id (null = root) and store mime, size_bytes, and blob_name.Must
FR-MODEL-03Folder names shall be case-insensitively unique per parent among non-deleted siblings (enforced by a partial unique index on (parent_folder_id, lower(name))).Must
FR-MODEL-04Both entities shall support soft-delete via deleted_at and an orthogonal is_starred flag.Must
FR-MODEL-05File size shall be capped at 250 MB (262,144,000 bytes), enforced by a DB CHECK constraint in addition to API-layer checks.Must

8.2 Create Folder

POST /drive/folders with { name (≤255, no / \ NUL), parentId? }. The service resolves the parent within owner+agency scope (throws NotFound if the parent uid isn't owned by the caller), resolves a unique name (auto-suffixing " (1)", " (2)", … up to 50 attempts), and inserts a folder row with a fresh UUID.

IDRequirementPriority
FR-FOLDER-01The system shall create a folder under a caller-owned parent (or root) and return the shaped folder with HTTP 201.Must
FR-FOLDER-02On name collision within the parent, the system shall auto-suffix the name to the first free " (n)" (up to 50), else return PreconditionFailed.Must
FR-FOLDER-03Folder names shall reject /, \, NUL, and lengths > 255 at the validator.Must
FR-FOLDER-04The parent shall be resolved within the caller's owner+agency scope; unknown/foreign parents surface as NotFound.Must

8.3 Upload (Two-Phase, Direct-to-Azure)

Upload is two API calls around a direct browser→Azure PUT. Phase 1 POST /drive/files/sas with { name, declaredSize (1..250 MB), declaredMime (whitelist), folderId? } (rate-limited 30/min/user): the service resolves the target folder, pre-computes a non-colliding finalName, generates a file UUID, builds the blob path server-side as agencies/{serviceCenterUid}/owners/{ownerUid}/{fileUid}, obtains a 10-minute write SAS, and returns { fileUid, blobName, sasUrl, expiresAt, finalName, folderUid }. The browser then PUTs bytes directly to sasUrl with header x-ms-blob-type: BlockBlob. Phase 2 POST /drive/files/commit with { blobName, folderId?, name, declaredSize, declaredMime }: the service verifies the tenant prefix, HEADs the blob to confirm existence and actual size, sniffs the first 4 KB for MIME, re-resolves a unique name, and (in a transaction) inserts the file row and increments the storage counter.

FR-UPLOAD-01The system shall issue a 10-minute write SAS for a server-constructed blob path derived only from agency/owner/file UIDs (no user-supplied path segment).Must
FR-UPLOAD-02SAS issuance shall be rate-limited to 30 requests per minute per user (quota-burn guard).Must
FR-UPLOAD-03declaredMime shall be validated against the MIME whitelist and declaredSize bounded to 1..250 MB at Phase 1.Must
FR-UPLOAD-04At commit, the system shall reject any blobName not prefixed with agencies/{callerSC}/owners/{callerUid}/ (Forbidden) and require a non-empty, slash-free trailing segment.Must
FR-UPLOAD-05At commit, the system shall HEAD the blob and reject a size mismatch versus declaredSize, with a belt-and-suspenders 250 MB re-check.Must
FR-UPLOAD-06At commit, the system shall sniff the first 4 KB and reject when the sniffed MIME differs from the declared MIME; a null sniff (e.g. text/csv) accepts the whitelisted declared MIME.Must
FR-UPLOAD-07Commit shall insert the file and increment drive_storage_usage (+size, +1) atomically in one transaction, and return the file with HTTP 201.Must
FR-UPLOAD-08File bytes shall never transit the API server on upload (direct browser→Azure PUT via a fresh HTTP client).Must

8.4 Browse with Breadcrumbs

GET /drive/list with { view (my|shared-with-me|agency|starred|trash), parentId?, page, pageSize (≤200, def 50), sortField (name|updatedAt|sizeBytes), sortDirection (asc|desc) } returns { folders[], files[], breadcrumbs[], pagination, agencyUsage:{bytesUsed,fileCount} }. My Drive lists folders (unpaginated in-parent) and files (paginated) owned by the caller in the parent, builds breadcrumbs by walking parent_folder_id to root (capped at 25 hops against cycles), and includes agency usage. Other views dispatch to starred, trash, shared-with-me, and agency listings; unknown views return 501 NotImplemented.

FR-BROWSE-03Shared-with-me shall list items with a user share granted to the caller, excluding caller-owned items, with empty breadcrumbs.Must
FR-BROWSE-04Agency view shall list all agency-shared items in the agency regardless of owner.Must

8.5 Move, Rename & Star

PATCH /drive/folders/:uid and PATCH /drive/files/:uid accept at least one of { name, parentId/folderId, isStarred }. The item is fetched owner-scoped (else NotFound). parentId/folderId: null moves to root; undefined leaves it unchanged. Moving a folder is guarded by a recursive descendants CTE that rejects moving a folder into itself or a descendant (PreconditionFailed). Renames re-resolve a unique name in the destination, excluding self.

IDRequirementPriority
FR-EDIT-01The system shall allow the owner to rename, move (to another folder or root), and star/unstar folders and files.Must
FR-EDIT-02Folder moves shall be rejected if the destination is the folder itself or any of its descendants.Must
FR-EDIT-03Rename shall re-resolve a unique name at the destination (excluding the item itself).Must
FR-EDIT-04A PATCH with no recognised field shall be rejected by the validator.Must

8.6 Delete, Restore & Empty Trash

DELETE /drive/folders/:uid and DELETE /drive/files/:uid soft-delete (set deleted_at); folder deletion cascades via a recursive descendants CTE, setting deleted_at on all descendant folders and their files in a transaction. Restore (POST .../restore) clears deleted_at and re-parents to root if the original parent is still deleted (avoids orphaning inside a trashed parent). DELETE /drive/trash hard-deletes the caller's trashed files — best-effort Azure blob deletes (Promise.allSettled), DB delete, then decrement of the storage counter — and then hard-deletes trashed folders. The storage counter is unchanged on soft-delete/restore; only empty-trash decrements it.

IDRequirementPriority
FR-TRASH-01Soft-delete shall set deleted_at; folder deletion shall cascade deleted_at to all descendant folders and files atomically.Must
FR-TRASH-02Restore shall clear deleted_at and re-parent to root when the original parent remains deleted.Must
FR-TRASH-03Empty-trash shall hard-delete the caller's trashed files, best-effort delete their Azure blobs, and decrement drive_storage_usage by the summed bytes/count; then hard-delete trashed folders.Must
FR-TRASH-04Soft-delete and restore shall not change the storage counter (trashed items still count until hard-deleted).Must

9. Detailed User & System Flows

9.1 Create a Folder

  1. From "+ New → New folder", the user enters a name; VS360 calls POST /drive/folders with { name, parentId? }.
  2. The service authorises the VS360 context and resolves parentId to a DB id within the caller's owner+agency scope; a foreign/unknown parent surfaces as NotFound (never leaks). Null parent = root.
  3. resolveUniqueFolderName checks for a case-insensitive collision among non-deleted siblings; on collision it appends " (1)", " (2)", … up to 50 tries, matching the partial unique index.
  4. If 50 attempts fail, the service returns PreconditionFailed.
  5. Otherwise it inserts a drive_folders row with a fresh UUID v4 uid and returns the shaped folder (ownerIsMe computed) with HTTP 201.
  6. VS360 shows a success toast and invalidates the ["drive","list"] query so the new folder appears.

9.2 Upload a File to Azure (Two-Phase)

  1. The user drops a file onto the content area (or picks "+ New → Upload files"). VS360 reads the file's name, size, and MIME.
  2. Phase 1 — SAS: VS360 calls POST /drive/files/sas with { name, declaredSize, declaredMime, folderId? }. This call is rate-limited to 30/min/user.
  3. The service requires issuer.uid and serviceCenterInfo.uid, resolves the target folder within scope, pre-computes a non-colliding finalName, generates fileUid, and builds the blob path server-side as agencies/{serviceCenterUid}/owners/{ownerUid}/{fileUid} (no user string → path traversal impossible).
  4. The service requests a 10-minute write SAS from the Azure adapter and returns { fileUid, blobName, sasUrl, expiresAt, finalName, folderUid }.
  5. Direct PUT: VS360 uses a fresh axios instance (so NECX auth headers don't reach Azure) to PUT the raw bytes directly to sasUrl with header x-ms-blob-type: BlockBlob, reporting progress 0..1.
  6. Phase 2 — Commit: VS360 calls POST /drive/files/commit with { blobName, folderId?, name, declaredSize, declaredMime }.
  7. The service verifies blobName starts with agencies/{callerSC}/owners/{callerUid}/ (else Forbidden) and that the trailing segment (the file uid) is non-empty and slash-free (else BadRequest).
  8. It HEADs the blob to confirm existence and actual size, rejecting a size mismatch versus declaredSize and re-checking the 250 MB cap.
  9. It sniffs the first 4 KB (magic bytes) and rejects if the sniffed MIME differs from the declared MIME; a null sniff (e.g. text/csv) accepts the already-whitelisted declared MIME.
  10. It re-resolves a unique file name (concurrency race guard), then in a single transaction inserts the drive_files row (uid = fileUid) and increments drive_storage_usage by (+size, +1).
  11. It returns the committed file (HTTP 201); VS360 updates the floating progress widget and, once all files settle, shows one batch-summary toast.
  12. Edge cases: SAS expiry mid-upload → the PUT fails and the client surfaces an error for that file; a blob that fails size/MIME verification is rejected at commit and never recorded (leaving an orphan blob to be cleaned up); a name collision resolved between SAS and commit is handled by the commit-time re-resolve.

9.3 Browse & Navigate (My Drive with Breadcrumbs)

  1. VS360 holds parentId state and calls GET /drive/list?view=my&parentId=… (with page/sort options).
  2. The service lists all folders in the parent (unpaginated) and a page of files, both owned by the caller in that parent, and builds breadcrumbs by walking parent_folder_id up to root (capped at 25 hops to defend against cycles).
  3. The response includes folders[], files[], breadcrumbs[] (root-first, e.g. [{uid:null,name:"My Drive"}, …]), pagination, and agencyUsage.
  4. Double-clicking a folder row sets parentId to that folder's uid and re-lists; clicking a breadcrumb navigates to that ancestor.
  5. Single-clicking a row enters select mode and toggles selection; hover reveals a checkbox; shift-click extends a range; Escape or deselect-all exits select mode.
  6. Double-clicking a file opens it (see 9.5 download); double-clicking a folder navigates into it.
  7. Branches: switching the left-rail view re-issues list with the new view; unimplemented views return 501 and render as a graceful empty state; sorting by size on folders falls back to updated_at.

9.4 Move, Rename & Delete

  1. Rename: from the row kebab, the user edits the name; VS360 calls PATCH /drive/folders/:uid or /drive/files/:uid with { name }. The service fetches the item owner-scoped (else NotFound) and re-resolves a unique name at the destination (excluding self), then persists.
  2. Move: the user picks a destination in the Move dialog (a folder browser excluding the item being moved); VS360 sends { parentId } (folder) or { folderId } (file); null moves to root.
  3. For a folder move, the service runs a recursive descendants CTE and rejects moving the folder into itself or any descendant with PreconditionFailed; otherwise it updates the parent.
  4. Star/Unstar: VS360 sends { isStarred: !current }; the list is invalidated to reflect the change.
  5. Delete (soft): the user chooses "Move to Trash"; VS360 calls DELETE /drive/folders/:uid or /drive/files/:uid. For a folder, the service collects all descendant folder ids via a recursive CTE and sets deleted_at on those folders and all files within, in one transaction; for a file it sets deleted_at. The storage counter is unchanged.
  6. Restore: from Trash, the user restores an item; the service clears deleted_at, re-parenting to root if the original parent is still deleted.
  7. Empty Trash: the user confirms; VS360 calls DELETE /drive/trash. The service hard-deletes the caller's trashed files (best-effort Azure blob deletes via Promise.allSettled, then DB delete), decrements drive_storage_usage by the summed bytes/count, and then hard-deletes trashed folders.
  8. All of the above emit audit records.

9.5 Download a File

  1. The user opens a file (double-click or kebab → Download); VS360 calls GET /drive/files/:uid/download (with optional token for link access).
  2. The service resolves access: caller-owns, OR a direct/agency/cascading folder share, OR a valid share-link token (file-link matches, or folder-link ancestor of the file's folder). If none, it returns NotFound (identical to a missing file).
  3. On success it issues a 5-minute read SAS from the Azure adapter, emits a DOWNLOAD audit record, and returns { url, fileName } as JSON.
  4. VS360 (which cannot follow a redirect without dropping the Bearer header) receives the JSON and navigates the browser to url, downloading directly from Azure.

9.6 Generate a Share Link

  1. From the row kebab → "Get link", the user opens the Link dialog and selects an expiry (1d/7d/30d/never); VS360 calls POST /drive/links with { targetKind, targetUid, expiresIn }.
  2. The service verifies the caller owns the target, generates a 32-byte base64url token, stores only its SHA-256 hash and computed expires_at, and returns { uid, token, expiresAt }.
  3. VS360 displays the plaintext token exactly once with a copy button; the token is never retrievable again.
  4. The dialog lists active links (GET /drive/links?targetKind&targetUid, no tokens); the user can revoke via DELETE /drive/links/:uid (creator-only, sets revoked_at).
  5. A link holder later downloads via GET /drive/files/:uid/download?token=…; access is re-validated (hash match, non-revoked, non-expired) on every request.

9.7 Share with Users or the Whole Agency

  1. From the kebab → "Share", the user opens the Share dialog. To share agency-wide, they toggle "agency"; VS360 calls POST /drive/shares with { targetKind, targetUid, shareKind:'agency' }, creating a single row.
  2. To share with specific colleagues, the user selects same-agency staff via autocomplete; VS360 calls POST /drive/shares with { shareKind:'user', recipientUserIds:[…] }.
  3. The service verifies ownership, resolves recipient uids to same-agency users (excluding the caller), rejects when none are valid, and bulk-inserts with onConflict().ignore() (dedup via partial unique indexes).
  4. Existing shares are listed via GET /drive/shares?targetKind&targetUid (owner-only, hydrated names); a share is revoked via DELETE /drive/shares/:uid (granter, or the recipient for a user-share).
  5. Recipients see the item under Shared with me (user shares) or Agency Drive (agency shares); folder shares cascade read access to descendants.

9.8 View Storage Usage

  1. The rail and header always show the agency's bytesUsed (and item count) taken from agencyUsage in the latest list response.
  2. The user opens the Storage view; VS360 calls GET /drive/storage/breakdown.
  3. The service groups the caller's live files by MIME into 9 categories (images/videos/audio/documents/spreadsheets/presentations/text/archives/other), each with a label and colour, and returns non-empty categories plus a grand total {bytesUsed,fileCount}.
  4. VS360 renders an overall quota bar (total.bytesUsed / 15 GB, a display-only reference) and per-category bars.
  5. Note: the 15 GB figure is a UI constant only; the backend does not enforce it — usage is tracked, not capped (see Risks).

9.9 Copy a Shared File to My Drive

  1. For a file the caller does not own (ownerIsMe === false), the kebab offers "Save a copy to My Drive"; VS360 calls POST /drive/files/:uid/copy-to-my-drive.
  2. The service confirms read access (owned or shared), allocates a new uid and destination blob under the caller's prefix, and performs an Azure server-side copy (bytes never transit Express), polling to a 30 s deadline.
  3. In a transaction it inserts a new drive_files row at the caller's root with a unique name and increments the storage counter; it emits a COPY audit record and returns the new file (HTTP 201).

9.10 Search Files

  1. As the user types in the search box (debounced ~300 ms), VS360 calls GET /drive/search?q&view&limit=20 when q is non-empty.
  2. The service escapes LIKE metacharacters in q, runs ILIKE %q% on lower(name) scoped to the mapped view (my / shared-with-me / agency), and returns { folders[], files[] }.
  3. VS360 replaces the normal listing with the search results; clearing the box restores the browse view.

13. Non-Functional Requirements

13.1 Security & Privacy (Victim Safety)

  • Tenant isolation: every query is filtered by service_center_id; foreign-tenant rows surface as NotFound to avoid existence leaks. This is critical because stored files may include sensitive victim/survivor documentation.
  • Path-traversal prevention: blob paths are constructed server-side from agency/owner/file UIDs only; the filename never appears in the path, and the validator rejects /, \, NUL, and names > 255.
  • MIME anti-spoofing: a MIME whitelist at the API edge plus a magic-byte sniff at commit; a declared/sniffed mismatch is rejected.
  • Size cap: 250 MB enforced at the Joi validator, at commit (post-HEAD re-check), and by a DB CHECK constraint.
  • Least-exposure transfer: file bytes move directly browser↔Azure via short-lived SAS (10-min write, 5-min read); a fresh HTTP client is used for the Azure PUT so NECX Bearer tokens never reach Azure.
  • Share-link secrecy: tokens are 32-byte random values stored only as SHA-256 hashes; the plaintext is revealed exactly once; expiry and revocation are re-checked on every access.
  • No existence leaks: unauthorised or missing items both return NotFound.

13.2 Performance & Scalability

  • SAS issuance target < 300 ms; commit target < 500 ms (excludes the direct Azure PUT, which scales with file size and the client's link).
  • Direct-to-Azure transfer keeps large-file bandwidth off application servers.
  • Listing uses composite indexes; search uses a GIN trigram index; folder tree walks (breadcrumbs, descendants, ancestor-for-share) use bounded recursive CTEs (breadcrumbs capped at 25 hops).
  • Copy uses Azure server-side copy (bytes never transit Express), polled to a 30 s deadline.

13.3 Availability & Reliability

  • Best-effort blob operations on cleanup (empty-trash uses Promise.allSettled so one failed blob delete doesn't abort the batch).
  • Metadata mutations that must stay consistent with usage (commit, copy, empty-trash) run in DB transactions with the counter update.
  • Commit-time verification (HEAD size + MIME sniff) prevents recording corrupt or mis-declared uploads.

13.4 Auditability

  • A logAudit record is written for download, soft-delete, restore, share create/revoke, link create/revoke, copy, and empty-trash, capturing action/resource constants (integrated with the platform's ActionLogs audit trail).

13.5 Data Lifecycle & Retention

  • Items are "live" (deleted_at NULL) or "trashed" (deleted_at set); starred is orthogonal. Trashed items remain recoverable until hard-deleted.
  • The UI states a 30-day Trash retention, but only manual empty-trash currently hard-deletes and decrements usage; an automatic retention purge worker is a planned future phase (see Risks/Out of Scope).

13.6 Localization & Accessibility

  • The VS360 UI follows the platform's Mantine component conventions; localization and accessibility inherit VS360 web standards. No Drive-specific localization requirements beyond standard UI strings.

13.7 Compliance

  • The audit trail, tenant isolation, and access controls support the platform's overall compliance posture for handling sensitive victim data. Files stored here are agency operational records, not case-linked signed documents (which live in the E-Sign / Document Center).

14. Analytics, Reporting & Notifications

14.1 Storage Analytics

  • Agency usage meter: every list response carries agencyUsage:{bytesUsed,fileCount}, rendered in the left rail and header, and against a 15 GB display reference.
  • Storage breakdown: GET /drive/storage/breakdown returns per-category (9 MIME buckets) {label,color,bytesUsed,fileCount} plus a grand total, rendered as an overall bar and per-category bars in the Storage view.

14.2 Notifications

  • None. NECX Drive emits no in-app, socket, FCM, or email notifications. All user feedback is via local VS360 toasts (upload progress, per-file and batch summaries, success/error on actions). Views not yet implemented return 501 and render as a graceful empty/"coming soon" state.
