2. Executive Summary

NECX is a coordinated victim-services platform that connects survivors of crime with the agencies (service centers) and advocates who help them. The atomic unit of help delivery is the case — technically a service claim (service_claims): one victim's request for one published service, tracked from intake through resolution. Case Management is the module that governs the entire life of that record: how it is created (by a victim in the mobile app or by an advocate in the VS360 web app), how it moves through its status state machine (pending → approved → resolved, or pending → rejected), and everything an advocate does around it — capturing intake answers, writing internal notes, publishing victim-facing updates, uploading documents, chatting with the client in real time, referring the case to a partner agency, and closing it out with the grant-reporting (PMT) data funders require.

This document is scoped entirely to Case Management. It specifies the case data model, the state machine and its side effects, the case list and its filters, every case-detail tab (demographics, intake questions/answers, notes, published updates, documents, referrals, chat, timeline, and the PMT snapshot), the two creation paths, the approve/reject/resolve actions, the link-client / anonymous-case flow with login codes, file uploads, question-answer editing, the distinction between internal notes and published updates, how multiple services are consolidated under one client, the PMT snapshot lifecycle, claim statistics/KPIs, and every notification and socket event the module emits.

Case Management matters because it is where victim safety, dignity, and privacy are operationalized. A case can begin anonymously (advocate-created, no identifying contact) and only later be claimed by the victim via a one-time login code — so that a survivor never has to expose an email or phone number before they are ready. Internal notes are never shown to the victim; only explicitly *published* updates are. Cross-agency data sharing happens only with the victim's consent and only for the fields an advocate elects to share. Every mutation is written to an immutable audit trail. Case Management is therefore both the operational core of day-to-day advocacy and the compliance backbone for federal/state grant reporting.

3. Product Overview & Vision

3.1 Problem Statement

Victim-services agencies juggle intake forms, case notes, funder reporting spreadsheets, and fragmented communication channels, often across disconnected tools. Survivors, meanwhile, have no single, safe place to request help, see progress, and communicate with an advocate without prematurely exposing identifying information. The result is duplicated data entry, lost context when a client is referred between agencies, inconsistent grant reporting, and friction that can deter a survivor from seeking help.

3.2 Product Vision

Case Management makes the service claim the single source of truth for a survivor's request for help. From one record an advocate can triage, communicate, document, report, refer, and resolve — while the survivor sees a safe, controlled view of the same case on their phone. Intake questions captured at claim time flow into the case detail; grant-reporting data is captured as a by-product of normal case work (the PMT snapshot) rather than as a separate chore; and a case can move between agencies (via referral) or be consolidated under one client account (via link-client) without re-keying.

3.3 Where This Sits in the NECX Ecosystem

Case Management is served by the single unified NECX backend (necx-backend_v2) through its tri-router pattern and consumed by multiple clients:

  • Victim mobile app (Flutter, issuer type APP) — victim-initiated case creation, viewing published updates, chat, rating, and claiming anonymous cases via login code (/mobile/service-claims/*).
  • VS360 web app (React 18 + Vite + Mantine, issuer type VSMS) — the advocate/caseworker workbench: case list, create-case, case detail with all tabs, approve/reject/resolve, notes/updates, referrals, PMT (/web/service-claims/*).
  • NECX Admin console (issuer type NECX) — read-only oversight of cases across agencies (/admin/service-claims/*; list + details only, no mutation).
  • Adjacent modules — Referrals & Partnerships (cross-agency case cloning), Reports/PMT (grants, questionnaires, generated reports), Chat/Messaging (case chat threads), Dashboard (case KPIs), Notes Center (agency-wide view over claim_notes).

4. Goals, Objectives & Success Metrics

4.1 Goals & Objectives

  • Provide a single, authoritative case record (the service claim) covering intake, communication, documentation, reporting, and resolution.
  • Enforce a clear, auditable status state machine (pending → approved → resolved; pending → rejected) with correct side effects at each transition.
  • Enable both victim-initiated and advocate-created cases, including safe anonymous cases that a victim can later claim via a one-time login code.
  • Keep internal case work (notes, PMT data) strictly separated from what the victim sees (published updates, chat).
  • Capture grant-reporting (PMT) data as a by-product of case work and preserve it through the case lifecycle.
  • Give advocates fast triage through a sortable, filterable case list with unread-message prioritization.
  • Surface accurate case KPIs to advocates, agency admins, and the reporting pipeline.

5. Scope

5.1 In Scope

  • The service-claim (case) data model, including uid format, status, login-code fields, ratings, and unseen-update flag.
  • Case lifecycle and status state machine (pending, approved, rejected, resolved) plus the synthetic requested state surfaced from Request-to-Connect.
  • Case list with issuer scoping, filters, header stat chips, sorting (unread-first), and CSV/XLS export.
  • Create-case: victim-initiated (claimService from mobile) and advocate-created (shouldCreateUser from web), including duplicate guards and advocate auto-approve.
  • Approve / reject / resolve actions and every side effect (published updates, chat-thread open/archive, stats attribution, PMT deletion on reject, referral status flip, notifications).
  • Case detail tabs: Details (client answers + PMT questions), Chat, Updates, Notes, Files, Referrals, plus demographics, appointments/reminders context, and a case timeline view.
  • Link-client (attach contact to an anonymous case) and the login-code issuance/redemption flow.
  • File uploads on a case (≤ 5 files, context SERVICE_CLAIM) and the file translator tool.
  • Intake question/answer capture and full-replace editing.
  • Internal notes (claim_notes) vs. published updates (claim_updates) — creation, editing, deletion, and their differing visibility/notification behavior.
  • Consolidating multiple services/cases under one client via link-client; the fact that each service request is its own case.
  • PMT snapshot creation, questionnaire linkage, editing, and deletion-on-reject.
  • Claim statistics and KPIs (per-claim attribution via service_claim_stats; agency KPIs via getAgencyCaseStats / dashboard).
  • RBAC for cases (claims.view/add/edit/delete/view_all) and issuer scoping (APP / VSMS / ADMIN).
  • Every case-related notification and socket event (VS360 staff side and victim side).

5.2 Out of Scope

  • The Services & Categories catalog (publishing services, category taxonomy) — covered only where it feeds intake questions and case creation. See the Victim Services PRD.
  • Inquiries (anonymous pre-case message threads) except as a source of prefill data for Create Case.
  • The full Referrals & Partnerships lifecycle (partnership creation/accept/decline, cross-agency chat) — covered only at the case's referral integration points. See the Referrals & Partnerships PRD.
  • Grant CRUD, questionnaire authoring, generated reports, and the AI report builders — covered only where the PMT snapshot and per-claim questionnaire answers touch a case. See the Reports & PMT PRD.
  • Hotline, Shelter, Communities, and general Chat internals — referenced only where they seed or attach to a case.
  • Dashboard analytics screens beyond the case KPIs they compute.
  • A one-click "convert inquiry → case" action — not implemented (see Open Questions).

7. System Context & Architecture

7.1 Tech Stack

  • Backend: Node.js + Express (ESM), Objection.js/Knex over PostgreSQL, Socket.IO, Redis, BullMQ (email/jobs). Module root src/apps/victim-services/service-claim/ (with notes.* and updates.* sub-modules).
  • VS360 web: React 18 + Vite, Mantine v7, Redux Toolkit + redux-persist, TanStack React Query v4, react-router-dom v6, TipTap (rich text), socket.io-client. Case services in src/services/claims.js.
  • Victim app: Flutter (GetX), consuming /mobile/service-claims/*.
  • NECX Admin: consumes /admin/service-claims/* (read-only).

7.2 Tri-Router & Guards

Every case route file builds three sub-routers by URL prefix and RBAC context (allowOnly reads req.issuer.roleInfo.context), all wrapped with rateLimiter(rateLimiters.standard):

Sub-routerPrefixGuardConsumer
adminRouter/admin/service-claims/*allowOnly('ADMIN')NECX Admin (list + details only)
webRouter/web/service-claims/*allowOnly('VS360')VS360 web / advocate (VSMS staff)
mobileRouter/mobile/service-claims/*none (auth only)Victim mobile app (APP)

issuer.type ∈ APP (victim), VSMS (agency staff), NECX (admin). Case services branch heavily on issuer.type. Requests are AES-response-encrypted (clients decrypt); VS360 also sends X-Netlify-Secret.

7.3 Backend Communication

VS360 axios factory adds Authorization: Bearer <access>, x-refresh-token, X-Netlify-Secret, X-Environment; responses are decrypted client-side. On 401 the interceptor logs out; 403 renders UnauthorizedAccess. Case reads/writes use React Query keys that include user.roleInfo.permission.claims.view_all so the cache differs for admins vs. workers.

7.4 Real-Time & Notifications

req.io is threaded through case controllers. Two notification services fire per case event: VS360NotificationService (staff/socket via vs360-user-<uid> room + service_center_notifications) and AppNotificationService (victim via app-notification-room + victim_notifications + FCM). All notification calls are wrapped defensively — failures are logged, never thrown, and never break the main transaction. Specific channels/events are enumerated in Sections 11 and 14.

7.5 Tenancy

Agency-scoped data is filtered by issuer.serviceCenterInfo.id; a VSMS user only ever sees/edits their own agency's cases. APP issuers are scoped to issuer.id (their own cases). The claims.view_all flag further narrows a VSMS user to only pending/self-actioned cases when absent. Cross-agency isolation (CJIS-style) is enforced when resolving agency-scoped claims and notes.

8. Functional Requirements

8.1 Case (Service Claim) Core

A case is a service_claims row: a victim's request for one service. It carries a human-readable uid (CL<yy><month><uuid6>), serviceId, victimId (FK app_users), status, rating, hasUnseenUpdates, and login-code fields (loginCode, codeSentAt, codeUsedAt, codeLastUpdatedAt). Advocate-created cases (shouldCreateUser) auto-approve.

9. Detailed User & System Flows

9.1 Victim-Initiated Case Creation (claimService, transactional)

  1. Victim opens a service in the mobile app; the service detail fetch increments viewCount and returns claim-state for this victim.
  2. Victim answers the service's intake questions (fill-in-the-blank / multiple-choice / true-false; required questions enforced) and optionally adds a statementFromVictim.
  3. Client POSTs /mobile/service-claims with serviceId, questionAnswers[], statementFromVictim?.
  4. The server resolves victimId = issuer.id (APP).
  5. Duplicate guard: if an ongoing pending/approved claim already exists for (service, victim), the request is rejected.
  6. Create the claim with uid = CL<yy><month><uuid6>; status = pending (victim path is not auto-approved).
  7. Insert claim_question_answers for the claim.
  8. Create a service_claim_stats row (no approval attribution yet).
  9. Compute isNewClient; create the pmt_report snapshot with demographics.
  10. Emit claim-request to caseworkers assigned to the service's category (vsms_user_service_categories), inserting service_center_notifications and emitting socket claim-request to vs360-user-<uid>.
  11. Return { newClaim } to the app; the case now appears in the agency's "My Cases" as a New Case.

9.2 Advocate-Created Case (VS360 Create Case, shouldCreateUser)

  1. Advocate clicks "Create Case" and steps through: (1) Required Info — pick a service (limited to assigned categories) and enter First/Last name; case is anonymous by default. Optionally check "Add Contact & Notification Details" to reveal Contact Method (Email/Phone) and Preferred Notification Method (otpChannel: EMAIL/SMS/NONE).
  2. Step 2 — Optional Demographics: age range, gender (+other), race/ethnicity (+other), special classification (+other), state → city (cities lazy-load).
  3. Step 3 — Service Questions: dynamic per selected service; required questions may be individually "Bypass Requirement"-checked.
  4. On submit, VS360 calls POST /service-claims with serviceId, userData (name/contact/demographics), shouldCreateUser:true, questionAnswers[].
  5. Backend (advocate path): find an existing user by email/phone; if found, link the case to that account. Otherwise create a temp anonymous user (anonymousUserCode, isVerified:false).
  6. Create the claim; status = approved (advocate auto-approve because the acting principal is staff, not an app user).
  7. Insert answers; create service_claim_stats with approvedBy/At set (auto-approved); compute isNewClient; create the PMT snapshot.
  8. If a new user was created and otpChannel ≠ NONE, generate a login code and send the code notification.
  9. Because this is advocate-created, the claim-request caseworker notification is NOT emitted (the case is already approved).
  10. Return { newClaim, user?, code? }. VS360 shows "Case Added to Existing Account" (existing user) or opens CodeDisplayModal with the login code (new anonymous/coded user).
  11. Because the case is auto-approved, its SERVICE chat thread and "Approved" published update follow the same side effects as the approve flow (see 9.3) where applicable.

9.3 Approve a Case (pending → approved)

  1. Advocate opens a pending case; header shows Approve / Reject (or Accept/Decline referral when referralSegment === 'referral_received').
  2. Advocate clicks Approve and confirms (modals.openConfirmModal).
  3. VS360 calls PATCH /service-claims/approve/:uid.
  4. Backend patches status = approved.
  5. Backend creates a published claim_updates row titled "Approved".
  6. Backend inserts a new SERVICE chat_thread (victim ↔ caseworker), enabling the Chat tab.
  7. Backend sets service_claim_stats.approved_by/at (caseworker attribution).
  8. If this case is a referred (cloned) case, backend patches the source referral referralStatus = accepted (AgencyReferrals.patch(...).where('referredCaseId', claim.uid)).
  9. Backend notifies the victim: insert victim_notifications SERVICE_CLAIM_APPROVED + FCM + socket service-claim-action-for-<victimUid> and community-status-notification-<victimId>.
  10. VS360 decrements the claims notification badge; the case moves to Approved and its status-gated tabs enable.

9.4 Reject a Case (pending → rejected)

  1. Advocate opens a pending case and clicks Reject; confirms.
  2. VS360 calls PATCH /service-claims/reject/:uid.
  3. Backend patches status = rejected.
  4. Backend sets service_claim_stats.rejected_by/at.
  5. If referred, backend patches the source referral referralStatus = declined.
  6. Backend deletes the case's pmt_report snapshot.
  7. Backend notifies the victim (rejection notification via AppNotificationService + FCM + socket).
  8. The case moves to Rejected (read-only); most tabs are disabled.

9.5 Resolve a Case (approved → resolved)

  1. Advocate opens an approved case; header shows Refer (if eligible) and Mark as Resolved.
  2. Advocate clicks Mark as Resolved; VS360 opens SurveyModalForm to complete PMT/questionnaire data.
  3. On submit, VS360 calls PATCH /service-claims/resolve/:uid with pmtValues and threadUid.
  4. Backend patches status = resolved.
  5. Backend creates a published claim_updates row titled "Resolved".
  6. Backend sets service_claim_stats.resolved_by/at.
  7. Backend archives the case's chat thread (identified by threadUid).
  8. Backend notifies the victim (CLAIM_UPDATE/resolution notification + FCM + socket update-<receiverUid> / service-claim-action-for-<victimUid>).
  9. The case moves to Resolved; it sorts low in the list.

9.6 Link Client to an Anonymous Case (linkClientToCase)

  1. Advocate opens an anonymous case (no email/phone) and clicks "Link Client Contact" in the right rail.
  2. In LinkClientModal, the advocate enters an email or phone and selects otpChannel.
  3. VS360 calls PATCH /service-claims/link-client/:claimUid.
  4. Backend validates the channel/contact; rejects if the user already has contact info, or if a verified account already uses that email/phone.
  5. Backend sets the contact + otpChannel on the client.
  6. Backend reuses an existing loginCode if present, else generates a new one; sets codeSentAt.
  7. Backend sends the code notification (email/SMS) and logs ADVOCATE_LINKED_CLIENT.
  8. Backend returns a code where applicable; VS360 shows it in CodeDisplayModal.
  9. The victim redeems the code on the mobile app (send-otp-for-code → verify-otp-for-login-code → login-with-code / set-password-first-login), activating the account and gaining access to this and any other linked cases.

9.7 Login-Code Redemption (Victim Claims an Anonymous Case)

  1. Victim receives the login code (email/SMS) generated at case creation or link-client.
  2. Victim installs/opens the mobile app and chooses "log in with code."
  3. Client POST send-otp-for-code → server delivers an OTP to the code's channel.
  4. Client POST verify-otp-for-login-code → server activates the account (marks verified).
  5. Client POST login-with-code and/or set-password-first-login to establish credentials.
  6. Server records codeUsedAt on the case; the case's "Login Code" badge flips to Used in VS360.
  7. The victim now sees the case, its published updates, and can chat; all cases linked to the client roll up under the account.

9.8 Upload Files to a Case

  1. Advocate opens the Files tab and drops up to 5 files.
  2. VS360 calls POST /service-claims/upload-files/:uid (multipart).
  3. Backend writes each buffer to uploads/claim, uploads via fileService to remote storage, and returns remote URLs.
  4. Backend inserts a FileUploads row per file with context = SERVICE_CLAIM.
  5. The Files tab refreshes, listing case files alongside chat-attached files.
  6. Optionally, the advocate opens a file in the File Translator (POST /tool/translate/file).

9.9 Edit Intake Question Answers

  1. On an approved case, advocate opens the Details tab → Client Answers → Edit (EditClaimQuestionsModal).
  2. Advocate revises answers; VS360 calls PATCH /service-claims/claims/:uid/questions with the full answer set.
  3. Backend deletes all existing claim_question_answers for the claim, then re-inserts the provided set (full replace).
  4. The read side re-merges the service's full question list with answers so any still-unanswered questions render id:null.
  5. Edit controls remain disabled unless the case is approved.

9.10 Add / Edit / Delete an Internal Note

  1. Advocate opens the Notes tab and clicks New Note; enters title, TipTap rich text, and selects a noteType.
  2. VS360 calls POST /service-claims/new-note/:claimUid; backend inserts a claim_notes row with createdBy = issuer.id.
  3. To edit, PATCH /service-claims/update-note/:noteUid (sets updatedBy); to delete, DELETE /service-claims/delete-note/:noteUid.
  4. Notes are never sent to the victim and never trigger a victim notification; they are additionally visible in the agency-wide Notes Center over the same claim_notes table.

9.11 Create / Publish / Edit / Delete a Victim-Facing Update

  1. Advocate opens the Updates tab, clicks Create Update, enters a title + rich text, and toggles "notify client" (publish).
  2. VS360 calls POST /service-claims/new-update/:claimUid.
  3. If published (isPublished=true), backend notifies the victim: CLAIM_UPDATE victim_notifications row + socket update-<receiverUid> + FCM; sets hasUnseenUpdates=true.
  4. Editing via PATCH /service-claims/update-update/:updateUid: an unpublished→published transition fires the same victim notification (publish trigger).
  5. Deleting via DELETE /service-claims/delete-update/:updateUid.
  6. The mobile app lists only isPublished=true updates (GET get-updates/:claimUid filtered for APP).

9.12 Victim Marks Updates Seen

  1. Victim opens a case with hasUnseenUpdates=true in the mobile app.
  2. Client calls PATCH /service-claims/claim-update/seen/:uid.
  3. Backend clears hasUnseenUpdates; the case's unseen indicator disappears.

9.13 Case Chat (Client ↔ Advocate)

  1. Chat is available only after approval (the SERVICE chat thread is created on approve, or on advocate auto-approve).
  2. Advocate opens the Chat tab; VS360 fetches GET /chat-services/get-all-chats/:uid (paged).
  3. Either party sends a message (POST /chat-services/send-message/:uid, multipart with optional attachments).
  4. New messages generate SERVICE_NEW_MESSAGE notifications; the case list bubbles cases with unread messages to the top.
  5. On resolve, the thread is archived (read-only history retained).

9.14 Refer a Case to a Partner Agency (integration point)

  1. On an approved case with accepted partnerships and no active referral, advocate clicks Refer.
  2. In ReferAgencyModal, advocate searches accepted partners (both agencies acceptingReferrals=true), picks a target service, chooses which case data to share (Case Notes, Files, Chat History, Updates, Client Answers, PMT Demographics), and writes a message.
  3. On confirm, a referral is created (consentStatus=pending, referralStatus=pending, referredCaseId=null); the single-active-referral guard returns 409 if one is already active.
  4. The victim consents (or rejects) on the mobile app. On consent, a sanitized case snapshot is built per the sharing flags and the case is cloned into the target agency (new service_claims + mirrored pmt_report + baseline service_claim_stats, status pending).
  5. The receiving agency approves/rejects the cloned case; that approve/reject flips the source referral referralStatus to accepted/declined (see 9.3 step 8 and 9.4 step 5).
  6. The Referrals tab on both the source and cloned cases reflects the sent/received referral and the shared snapshot sections.

9.15 Consolidate Multiple Cases Under One Client

  1. A client has several separate cases (each its own service claim), possibly created anonymously at different times.
  2. An advocate links the client's contact to each anonymous case (link-client) using the same email/phone.
  3. Because the cases resolve to one client account, the victim — after redeeming the login code — sees all their cases together in the mobile app.
  4. There is no case-merge operation; consolidation is purely by client account, and cross-agency handoff remains referral-only.

9.16 PMT Snapshot Lifecycle

  1. At claim creation, the backend computes new-vs-returning client and inserts a pmt_report row with demographics captured at intake (age, gender, race/ethnicity, special classification).
  2. During case work, applicable questionnaire(s) (matched by the service's category and linked to an active grant) are resolved via GET /reports/pmt-questionnaires/claim/:claimId; answers are submitted via POST /reports/pmt-questionnaires/claim/:claimId/answers.
  3. PMT demographic fields can be corrected via PATCH /reports/pmt-report/:uid (agency-scoped; 403 if the row's claim is not in the issuer's agency).
  4. On resolve, final pmtValues from SurveyModalForm are persisted.
  5. On reject, the pmt_report snapshot is deleted.
  6. When a case is cloned by a referral, a mirrored pmt_report row is created in the target agency (demographics copied from victimInfo).

9.17 View & Filter the Case List

  1. Advocate opens "My Cases"; VS360 calls GET /service-claims with page, pageSize:20, and current filters; the query key includes claims.view_all.
  2. Backend scopes results: VSMS → agency cases (pending/self-actioned only without view_all); APP → own cases; ADMIN → cross-agency read.
  3. Header chips show New Cases, Approved Cases, and Referred Cases; response also returns requestsToConnect[].
  4. Advocate filters by search (client name or Case ID), Category (limited to assigned categories), and Status; refresh re-queries.
  5. Cases render as cards, sorted unread-first (matched against SERVICE_NEW_MESSAGE notifications), then by status priority, then recency.
  6. Advocate clicks a card → navigate to /claims/:uid (case detail), passing the claim in route state.
  7. Advocate optionally exports the list to CSV/XLS via a pageless fetch.

9.18 Quick Case-Status Update from Dashboard

  1. On the dashboard, advocate uses StatusUpdateModal.
  2. VS360 calls PATCH /dashboard/cases/:uid/status with {status, reason} (guarded by claims.edit).
  3. Backend validates status against the whitelist (pending, approved, rejected, resolved, in_progress, pending_review, awaiting_client, pending_court_date, closed), verifies the case belongs to the issuer's agency (403 otherwise), and patches status + statusReason.
  4. Note: this dashboard path allows extended workflow statuses beyond the core state machine and does not run the approve/reject/resolve side effects — see Open Questions.

9.19 Rate a Resolved Case (Victim)

  1. After resolution, the victim opens the case in the mobile app.
  2. Client calls PATCH /service-claims/rating/:uid with a rating value.
  3. Backend patches rating on the case; the value feeds getAgencyCaseStats average rating and dashboard provider rating.

9.20 Request-to-Connect Precursor

  1. A victim browsing a service taps "request to connect."
  2. Client calls PUT /victim-services/request-to-connect/:uid.
  3. Backend dedupes (409 if already requested), emails the service center, and records the request.
  4. The request surfaces in the agency case list as requestsToConnect[] and appears to the app as a synthetic requested state; an advocate follows up (e.g., by creating a case).

13. Non-Functional Requirements

  • Victim safety & privacy (paramount). Cases may be fully anonymous (no email/phone) until the victim chooses to claim them via login code. Internal notes (claim_notes) are never exposed to the victim. Only explicitly published updates and case chat are victim-visible. Cross-agency sharing occurs only with victim consent and only for the specific fields an advocate elects to share.
  • Security. All endpoints require authentication; VS360/ADMIN routes require the context gate plus fine-grained claims.* permissions. Responses are AES-encrypted; VS360 additionally requires X-Netlify-Secret. Agency isolation prevents any VSMS user from reading/writing another agency's cases.
  • Auditability. Every case create/update/delete writes an action_logs row (action, referenceType, actionDescription{source,ip,url,context}); state transitions additionally attribute the acting caseworker in service_claim_stats. Login-code linking logs ADVOCATE_LINKED_CLIENT; referral consent logs REFERRAL_CONSENT_GRANTED/DENIED.
  • Performance. Case list and detail are paginated; write routes are rate-limited (rateLimiters.standard, ~10/5s). React Query caches reads with keys that vary by claims.view_all.
  • Availability & resilience. Case creation and state transitions run in DB transactions (withTransaction); notification/socket calls are wrapped defensively so a delivery failure never rolls back or breaks the primary case operation.
  • Data integrity. One-open-claim-per-(service,victim) guard prevents duplicate victim-initiated cases; answer editing is an atomic full replace; PMT snapshot is created on creation and deleted on reject to keep grant data consistent with case state.
  • Compliance (grant/PMT reporting). The PMT snapshot and per-claim questionnaire answers are captured as a by-product of case work to support VOCA/VAWA-style federal/state reporting.
  • Localization & accessibility. Victim-facing strings support the platform's multi-language static translations; VS360 UI follows Mantine accessible components.

14. Analytics, Reporting & Notifications

14.1 Case KPIs & Dashboards

  • Per-claim attribution in service_claim_stats (approved_by/at, rejected_by/at, resolved_by/at).
  • getClaimStats aggregate counts (new = pending, approved = approved), scoped by claims.view_all.
  • ServiceClaims.getAgencyCaseStats — completion rate, unique/new/returning clients, top-5 services, average rating.
  • Dashboard KPIs (GET /dashboard/kpi-data) — new/completed/total claims, average response time (approved_at − created_at), repeat-victim rate, service-provider rating, with month-over-month trend badges.
  • Cases-needing-attention widget uses GET /service-claims?statusNotIn=resolved,rejected.

14.2 Notifications & Socket Events (Case-Specific)

Staff side (VS360NotificationService, room vs360-user-<uid>):

  • New victim-initiated (non-advocate) case → emitServiceClaimNotification → service_center_notifications + socket claim-request to caseworkers assigned to the service's category.
  • New case chat message → SERVICE_NEW_MESSAGE notification (drives unread-first sorting in the case list).

Victim side (AppNotificationService + FCM, room app-notification-room):

  • Approve → victim_notifications SERVICE_CLAIM_APPROVED + FCM + socket service-claim-action-for-<victimUid> and community-status-notification-<victimId>.
  • Reject → victim rejection notification + FCM + socket.
  • Resolve → resolution/CLAIM_UPDATE notification + FCM + socket (service-claim-action-for-<victimUid> / update-<receiverUid>).
  • Published update (or unpublished→published) → CLAIM_UPDATE + socket update-<receiverUid> + FCM; sets hasUnseenUpdates.
  • Login-code delivery → email/SMS (Twilio) on creation and on link-client.
  • Request-to-connect → email to the service center.

All notification/socket calls are defensive: failures are logged and never break the primary case transaction.
